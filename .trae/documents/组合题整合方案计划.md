# 组合题整合方案计划

## Summary

目标：在现有本地训练考试系统中新增“组合题”能力，支持“一个大题干/材料 + 多个顺序固定的小题”，并保持现有普通题、练习模式、考试模式、历史考试记录完全兼容。

本方案采用“组合题容器”模式，而不是把组合题拆成松散的普通题。组合题在录题、编辑、练习、考试、结果展示中按“一个父题 + 多个子题”呈现；成绩统计按“小题”计分；题目管理列表按“父题一行 + 子题展开/查看”的方式展示。

首阶段只支持网页录入与网页答题，不要求同步支持 Excel 导入组合题。

## Current State Analysis

### 现有数据结构

- `app.py` 中的 `init_bank_db()` 当前只创建了 `questions` 和 `exam_records` 等表，题目模型是单表单题结构：[app.py](file:///d:/dev/family-exam-helper/app.py#L49-L98)
- `questions.type` 目前仅允许 `single_choice`、`multi_choice`、`true_false`、`fill_blank` 四种题型：[app.py](file:///d:/dev/family-exam-helper/app.py#L63-L75)
- 题目字段是面向单题设计：`content/options/answer/explanation` 全都直接挂在一条题目记录上。

### 现有管理流程

- 题目新增/编辑统一走 `_parse_question_form()`，只支持解析单题字段：[app.py](file:///d:/dev/family-exam-helper/app.py#L447-L487)
- 题目列表 `list_questions()` 直接查询 `questions` 表后渲染，没有父子层级概念：[app.py](file:///d:/dev/family-exam-helper/app.py#L336-L366)
- 题目录入页面 `question_form.html` 只支持一种题型对应一个答案区，没有子题编辑 UI：[question_form.html](file:///d:/dev/family-exam-helper/templates/question_form.html)

### 现有答题/判分流程

- 抽题逻辑 `_select_question_ids()` 返回的是普通题 `id` 列表：[app.py](file:///d:/dev/family-exam-helper/app.py#L951-L971)
- 练习页和考试页都假设“当前只显示一题”，模板按 `question.type` 分支渲染：[practice.html](file:///d:/dev/family-exam-helper/templates/practice.html)、[exam.html](file:///d:/dev/family-exam-helper/templates/exam.html)
- 判题逻辑 `_check_answer()` 仅接收单题 `qtype/correct_answer/user_answer`：[app.py](file:///d:/dev/family-exam-helper/app.py#L974-L992)
- 结果页 `result.html` 也是按“每个结果对象对应一道题”展示：[result.html](file:///d:/dev/family-exam-helper/templates/result.html)

### 当前限制结论

- 现有系统的核心假设是“题库里的每条题目都是独立题目”。
- 若直接把组合题硬塞进现有 `questions` 表，会导致录题、抽题、答题、判分、结果展示四处逻辑同时失真。
- 因此需要明确引入“父题/子题”模型，但必须采用增量兼容方式，不能破坏现有普通题流程。

## Proposed Changes

### 1. 数据模型升级

#### 1.1 扩展 `questions` 表承载父题

文件：`app.py`

在 `init_bank_db()` 的建表逻辑中扩展 `questions` 表：

- 将 `type` 的允许值增加 `composite`
- 对 `questions` 表新增以下字段：
  - `parent_id INTEGER DEFAULT NULL`
  - `sort_order INTEGER DEFAULT 0`
  - `stem TEXT DEFAULT ''`
  - `is_subquestion INTEGER DEFAULT 0`
- 约束语义：
  - 普通题：`type != 'composite'`，`parent_id IS NULL`，`is_subquestion = 0`
  - 组合父题：`type = 'composite'`，`parent_id IS NULL`，`is_subquestion = 0`，`stem/content` 用于存大题干
  - 组合子题：`type IN ('single_choice','multi_choice','true_false','fill_blank')`，`parent_id = 父题ID`，`is_subquestion = 1`

设计原因：

- 继续使用一张 `questions` 表，迁移成本最低
- 子题仍然复用现有四种题型与判题逻辑
- 父题只作为容器，不参与独立判分

补充索引建议：

- `idx_questions_parent_id(parent_id)`
- `idx_questions_parent_sort(parent_id, sort_order)`

#### 1.2 数据迁移策略

文件：`app.py`

在数据库初始化阶段补充轻量迁移逻辑：

- 启动时检测 `questions` 表是否缺少 `parent_id/sort_order/stem/is_subquestion`
- 若缺少则执行 `ALTER TABLE` 追加字段
- 检测 `type` 的历史约束无法直接修改时，采用“兼容读取 + 新库新约束”策略：
  - 对已有题库，允许应用层写入 `composite`，SQLite 检查若受旧约束限制，则通过迁移新表再复制数据
  - 推荐实现为一个显式的 `migrate_bank_schema(conn)`，由 `get_db()` 打开连接后调用一次

设计原因：

- 用户要求历史数据和考试记录完全兼容
- 不能要求人工重建旧题库

### 2. 后端模型与查询重构

#### 2.1 统一题目读取模型

文件：`app.py`

新增若干辅助函数：

- `get_question_with_children(db, qid)`：获取一条题，若是组合父题则附带已排序子题
- `list_question_rows(db, filters)`：列表页读取父题和普通题，默认不把子题当一级列表项返回
- `serialize_question_for_view(row, children=None)`：把数据库行转成模板友好的结构

行为要求：

- 题目管理列表默认只显示普通题和组合父题
- 如果需要展开子题，则通过额外查询按 `parent_id, sort_order` 拉取
- 题目详情接口 `/questions/view/<qid>` 对组合父题返回：
  - 父题基础信息
  - `children` 数组

#### 2.2 录题表单解析分流

文件：`app.py`

新增组合题专用解析函数：

- 保留 `_parse_question_form()` 作为普通题解析
- 新增 `_parse_composite_form(form)`：
  - 解析父题字段：科目、标签、难度、大题干、解析
  - 解析多个子题：`type/options/answer/content/explanation/sort_order`

保存策略：

- 新增组合题时：
  - 先插入父题 `type='composite'`
  - 再按顺序插入子题，写入 `parent_id` 和 `sort_order`
- 编辑组合题时：
  - 更新父题
  - 删除该父题原有全部子题
  - 按表单重建子题

设计原因：

- 当前项目规模小，编辑时“整体重建子题”比做差量更新更稳、更简单

#### 2.3 删除逻辑联动

文件：`app.py`

更新 `/questions/delete/<qid>`：

- 删除普通题时维持原逻辑
- 删除组合父题时联动删除全部子题
- 如果意外传入子题 `qid`，可直接删除该子题，或更稳妥地限制只允许从父题编辑页删除

推荐做法：

- 列表页默认不暴露对子题的独立删除入口
- 所有子题删除都通过编辑组合题时统一处理

### 3. 题目管理与录题界面

#### 3.1 题型选择增加“组合题”

文件：`templates/question_form.html`

调整题型下拉：

- 新增 `composite` 选项，文案如“组合题 / 材料题”
- 选择 `composite` 后切换到组合题专用编辑区域

组合题编辑区域包含：

- 父题信息：
  - 科目
  - 标签
  - 难度
  - 大题干/材料（TinyMCE）
  - 父题解析（可选）
- 子题列表编辑区：
  - 支持新增子题
  - 支持删除子题
  - 支持上下移动顺序
  - 每个子题可选择 `single_choice / multi_choice / true_false / fill_blank`
  - 每个子题拥有自己的题干、选项、答案、解析

实现建议：

- 保持当前页面单文件模式，用原生 JS 动态增删子题块
- 每个子题块使用命名约定，例如：
  - `sub_0_type`
  - `sub_0_content`
  - `sub_0_option_a`
  - `sub_0_answer`
  - `sub_0_explanation`
- 用隐藏字段 `sub_count` 或数组式命名辅助解析

#### 3.2 题目列表支持父题一行展示

文件：`templates/questions.html`

列表行为调整：

- 普通题保持现状
- 组合父题显示：
  - “组合题”类型徽标
  - 父题摘要
  - 子题数量，如“共 4 小题”
  - “展开子题”按钮
- 展开后显示子题清单：
  - 顺序号
  - 子题类型
  - 子题摘要

配套后端：

- `list_questions()` 返回父题的 `children_count`
- 如采用服务端渲染展开，可直接把 `children` 一并传给模板

### 4. 练习/考试流程改造

#### 4.1 抽题以“题组”为单位

文件：`app.py`

改造 `_select_question_ids()` 为“抽可展示单元”：

- 对普通题：返回其 `id`
- 对组合题：只返回父题 `id`
- 抽题集合默认排除 `is_subquestion = 1` 的子题

建议改名：

- `_select_question_unit_ids()` 或 `_select_parent_question_ids()`

这样练习/考试 session 中保存的是“展示单元 ID 列表”，而不是原先的裸题目列表。

#### 4.2 练习页整组同屏

文件：`app.py`、`templates/practice.html`

后端：

- 当当前题为普通题时，保持现有渲染结构
- 当当前题为组合父题时，向模板传：
  - `question`：父题
  - `children`：按 `sort_order` 排序的子题
  - `saved_answers`：以子题 `id` 为 key 的答案映射

前端：

- 页面顶部仍算“第 N 题 / 共 M 题”，其中 N/M 按展示单元计
- 父题区域显示大题干
- 父题下面按顺序渲染多个子题卡片
- 每个子题按各自类型渲染答案输入控件
- “查看答案”时可整组展开正确答案与解析

答案提交格式建议：

- 普通题继续使用原 `answer`
- 组合题提交：
  - `sub_answer_<child_id>` 多字段提交

session 存储建议：

- 现有 `practice_answers[qid] = answer`
- 扩展为：
  - 普通题：值仍为字符串
  - 组合题：值为对象，如 `{child_id: answer}`

#### 4.3 考试页整组同屏

文件：`app.py`、`templates/exam.html`

与练习页类似，但不提供“查看答案”按钮。

关键点：

- 用户交卷前可以在当前组内修改任一小题答案
- 组内答案一起保存到 session
- 下一题时整个题组一起前进

### 5. 判题、统计与结果页

#### 5.1 判题仍以子题为单位

文件：`app.py`

保留 `_check_answer()` 作为底层单题判题函数，不直接改复杂逻辑。

新增上层组合题判题辅助：

- `check_composite_answers(children, user_answers)`

返回：

- 每个子题的对错
- 组内正确数
- 组内总题数
- 组内得分

评分规则：

- 按小题计分
- 组合题父题本身不计分
- 总题数与正确数以所有子题数累计

#### 5.2 结果页支持“组合题结果块”

文件：`templates/result.html`

展示策略：

- 普通题：保持原样
- 组合题：
  - 结果列表里显示“第 N 题（组合题）”
  - 先展示父题材料
  - 再逐个展示子题：
    - 子题题干
    - 用户答案
    - 正确答案
    - 对错状态
    - 子题解析
  - 可额外显示“本组正确 3/4”

后端结果数据结构建议：

- 结果数组项统一为“展示单元”
- 普通题结构不变
- 组合题结构新增：
  - `children_results`
  - `group_correct`
  - `group_total`

#### 5.3 首页统计口径

文件：`app.py`、`templates/bank_home.html`

统计建议：

- 首页总题数继续显示“可作答小题总数”更符合用户理解
- 同时可附加一个组合题统计，如“其中组合题 12 组”

类型统计建议：

- 继续按真实作答题型统计子题数量
- 单独增加 `composite_groups` 用于展示组合题组数

### 6. 首阶段明确不做的内容

为控制首期复杂度，以下内容不纳入本次实现：

- Excel 导入组合题
- JSON 导出/导入的组合题结构升级
- 按小题跨组随机抽题
- 子题独立出现在题目列表一级筛选中
- 考试记录的明细报表重构

这些内容留作第二阶段扩展。

## Assumptions & Decisions

### 已确认决策

- 采用“组合题容器”而不是普通题分组
- 练习/考试都采用“整组同屏”
- 首阶段优先网页录题，不优先做 Excel 导入
- 计分按小题统计
- 题目管理列表按“父题一行 + 子题展开”展示
- 采用增量兼容策略，历史普通题和考试记录不迁移、不破坏

### 关键设计决定

- 复用 `questions` 表存父题和子题，不新增独立 `subquestions` 表
- 父题使用 `type='composite'`
- 子题继续沿用现有四种题型与 `_check_answer()`
- session 中允许答案值既可能是字符串，也可能是字典对象
- 组合题编辑采用“整组重建子题”的简单稳健策略

### 风险与注意项

- 现有 `questions.type` 带有 SQLite `CHECK` 约束，新增 `composite` 需要认真设计迁移逻辑
- 结果页、练习页、考试页目前都严重依赖“单题”假设，模板改动会比较集中
- JSON 导入导出与 Excel 导入首期不支持组合题时，需要在页面说明中明确告知

- 新建题库后确认 `questions` 表包含新增字段
- 旧题库打开后确认自动迁移成功，普通题可继续增删改查
- 新增一个组合题后确认父题和子题正确落库，`parent_id/sort_order/is_subquestion` 正确

### 管理功能验证

- 在 [question_form.html](file:///d:/dev/family-exam-helper/templates/question_form.html) 中创建一个包含 3 个子题的组合题
- 在 [questions.html](file:///d:/dev/family-exam-helper/templates/questions.html) 中确认只显示父题一行，并可展开看到子题
- 编辑组合题并调整子题顺序后，重新进入列表确认顺序生效
- 删除组合题后，数据库中父题和全部子题都被清理

### 练习/考试验证

- 练习模式抽到组合题时，页面同屏显示父题和全部子题
- 组合题内同时包含单选、判断、填空时，三类控件都能正常提交
- 练习模式点击“查看答案”后，整组正确答案和解析都能显示
- 考试模式交卷后，成绩按子题统计而非按题组统计

### 结果页验证

- [result.html](file:///d:/dev/family-exam-helper/templates/result.html) 能展示组合题父题和逐个子题结果
- 错误子题能显示用户答案、正确答案和解析
- 总正确数、总题数、正确率符合“小题口径”

### 回归验证

- 普通单选/多选/判断/填空的新增、编辑、练习、考试、结果页全部保持可用
- Excel 导入普通题不受影响
- JSON 导入导出普通题不受影响
